## Unreleased

### Fixes

- **`make ci` now actually mirrors the GitHub Actions CI workflow locally.** Two latent bugs in the local Makefile orchestration would cause `make ci` to fail with red e2e specs that pass in GH Actions: (1) `ci` depended on `lint typecheck test build linkcheck` in that order, so the `build` step ran AFTER `test` — meaning the e2e suite had no `out/` to serve and tests for static-only assets like `/resume.pdf` correctly 404'd; (2) the `e2e` and `a11y` make recipes invoked `npx playwright test` without `CI=true`, so the Playwright webServer block fell back to `npm run dev` (Next.js dev server) instead of `npm run start` (`serve out`) — meaning even with `out/` present, the dev server was being tested and its under-load timing flakiness manifested as `og:image` meta-tag assertion failures. GH Actions worked because every job step sets `CI: "true"` and the workflow has explicit `needs: build` ordering with artifact upload/download. Fix: reorder `make ci` to `lint typecheck unit build e2e a11y linkcheck` and prefix the `e2e`/`a11y` recipes with `CI=true`. Verified locally: `make ci` runs 130/130 e2e + 54/54 a11y + 0 lychee errors against a fresh `out/`. Failure mode now consistent between local and CI.
- **MDX frontmatter no longer leaks into rendered post bodies.** Every blog post (and every project case study) was rendering its `---YAML---` block as a thematic-break + paragraph + thematic-break at the top of the page. Root cause: `next.config.ts` called `createMDX({ extension: /\.mdx?$/ })` with no `remarkPlugins`. The MDX compiler had nothing telling it to strip the YAML; gray-matter only stripped it for the JS-side metadata parse. Verified by `grep -oE 'publishedAt:[^<]*' out/blog/*/index.html` showing the literal on every post. Fix: install `remark-frontmatter` and wire it through `createMDX({ options: { remarkPlugins: [["remark-frontmatter", { type: "yaml", marker: "-" }]] } })`. Turbopack requires the serializable `[name, options]` tuple form, not a function reference. New regression test in `tests/e2e/blog.spec.ts` asserts no post body contains the `publishedat:`, `slug: "`, or `excerpt: "` literals - 5/5 posts now pass.
- **Code blocks are readable in light and warm themes.** `prose-slate` ships a fixed dark-slate `<pre>` background (slate-800-ish) regardless of the active theme. The blog route layered `prose-code:text-[var(--color-text)]` on top, which set `<code>` text to slate-900 in the light + warm themes - dark text on dark background, contrast ~1.2:1, far below WCAG AA's 4.5:1. Fix: bind the four Tailwind Typography variables (`--tw-prose-pre-bg`, `--tw-prose-pre-code`, `--tw-prose-code`, plus their `invert` counterparts) to our theme tokens (`--color-surface-muted`, `--color-text`) inside a `.prose-slate { ... }` block in `app/globals.css`. Removed the now-redundant `prose-code:` modifier from `app/blog/[slug]/page.tsx`. Code blocks now adopt the active theme: light slate-100 background + dark text in light, slate-100 dark background + light text in dark, amber-100 background + dark text in warm. New regression tests in `tests/e2e/blog.spec.ts` compute the WCAG contrast ratio between `<pre>` background and `<code>` text via canvas-resolved sRGB and assert >= 4.5 across all three themes. Light theme contrast was 1.22:1 before the fix; full a11y suite (54 tests across three themes) still passes after.
- **`post-deploy-smoke` job no longer times out on `config.webServer`.** `playwright.config.ts` unconditionally declared a `webServer` block that runs `serve out` on port 3000 under `CI=true`. The `post-deploy-smoke` job sets `LIVE_URL` but never downloads the `next-build` artifact, so there was no `out/` to serve; Playwright waited the full 120s for `BASE_URL` to come up and failed before any spec ran. Fix: gate the `webServer` block on `process.env.LIVE_URL` being unset - when `LIVE_URL` is set, Playwright skips the local server and the smoke + security-headers specs hit the deployed Amplify URL directly (which they were already written to do via absolute URLs). Verified locally with the exact CI command: `LIVE_URL=https://main.d2f2x4ij8pxn6g.amplifyapp.com SECURITY_HEADERS_URL=... CI=true npx playwright test tests/e2e/post-deploy-smoke.spec.ts tests/e2e/security-headers.spec.ts --project=chromium-desktop` -> 11/11 passed in 2.7s. Failure mode now catalogued in `docs/DEPLOYMENT.md` "Common deploy failures".
