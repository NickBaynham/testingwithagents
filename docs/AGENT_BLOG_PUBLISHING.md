# Agent Runbook: Publishing a Blog Post

This document is for an agent (Claude, or any future operator) that has been asked to publish a new blog post. It is the operational runbook — what to write, what to update, what to verify, what to commit — so you do not have to rediscover the repo's conventions every time.

If you are a human reader, [`CONTENT_GUIDE.md`](CONTENT_GUIDE.md) is the high-level orientation. This file is the agent's step-by-step.

## What "publishing a post" means here

A blog post on `testingwithagents.com` is a single `.mdx` file under `content/blog/`. The site is statically generated by Next.js (`make build` → `out/`). Adding a post requires:

1. Authoring the MDX with the right frontmatter.
2. Updating every hardcoded reference to the previous "newest post" and the previous post count.
3. Adding a `CHANGELOG.md` entry.
4. Running `make ci` locally and proving it green.
5. Committing and pushing.

The repo has hardcoded post counts and slug lists in `tests/e2e/blog.spec.ts`. **Forgetting to update those is the #1 way to ship a red CI.** Use the checklist below.

## File map: where everything lives

| Path | Purpose |
| --- | --- |
| `content/blog/<slug>.mdx` | The post itself. Filename slug **must** match the `slug` in the frontmatter (loader enforces). |
| `lib/content/blog.ts` | Loader and Zod schema (`postFrontmatterSchema`). Schema violations fail `make build`. |
| `app/blog/page.tsx` | Blog index — auto-renders newest first. No edits needed for a new post. |
| `app/blog/[slug]/page.tsx` | Detail page — auto-renders. No edits needed. |
| `app/feed.json`, `app/rss.xml` | Feeds regenerate at build time. No edits needed. |
| `app/sitemap.ts` | Sitemap regenerates at build time. No edits needed. |
| `public/og/blog/<slug>.png` | Per-post OG image. **Auto-generated** by `scripts/generate-og-images.tsx` during `make build` (`prebuild` script). Do not author by hand. |
| `tests/e2e/blog.spec.ts` | **Has hardcoded post counts and slug lists. You MUST edit this.** See checklist below. |
| `CHANGELOG.md` | Unreleased / Added section. **You MUST add an entry.** |

## Frontmatter schema

Authoritative source: `lib/content/blog.ts` → `postFrontmatterSchema`. As of this writing:

```yaml
---
title: "Human-readable title, in quotes"
slug: "kebab-case-slug-matching-filename"
excerpt: "One-paragraph summary surfaced on the index, RSS, JSON Feed, and OG card. 2-4 sentences."
publishedAt: "YYYY-MM-DD"             # required; ISO date
updatedAt: "YYYY-MM-DD"               # optional
categories:
  - "Agentic Testing"                  # required, at least one, from POST_CATEGORIES enum in lib/content/blog.ts
tags:
  - "agents"                            # required, at least one; free-form kebab-case
  - "claude-code"
coverImage: "/blog/<slug>/cover.png"  # optional; served from public/
---
```

**Conventions for Test Commander walkthrough posts** (the long-form project-log style):

- `title`: descriptive, often two clauses separated by a colon. Example: `"Test Commander Phase 2: five commands, two new conventions, and the discipline of being domain-agnostic"`.
- `slug`: kebab-case, ≤ 60 chars where possible. Examples: `building-test-commander-plan-driven-tdd-session`, `test-commander-phase-2-domain-agnostic-by-design`.
- `excerpt`: 2–4 sentences. State what the post covers AND name the single most interesting lesson. The excerpt appears in RSS/JSON Feed/index cards, so it must be self-contained.
- `publishedAt`: today's date (the system date — check `date -u +%Y-%m-%d` if unsure).
- `categories`: for project-log posts, the canonical triple is `Agentic Testing`, `Test Automation`, `Project Logs`. Pick from the `POST_CATEGORIES` enum; the schema rejects unknown categories.
- `tags`: include `test-commander`, `session-walkthrough`, `phased-delivery`, `tdd`, `agents`, `claude-code`, `plugin-development` as the baseline. Add a new tag when a new theme deserves a label (e.g., `domain-agnostic` for the Phase 2 post).
- Do **not** set `coverImage` for project-log posts. The auto-generated OG image is sufficient.

## Body style (project-log walkthroughs)

The Test Commander build-session posts have a stable shape worth matching. Read [building-test-commander-plan-driven-tdd-session.mdx](../content/blog/building-test-commander-plan-driven-tdd-session.mdx) and [test-commander-phase-2-domain-agnostic-by-design.mdx](../content/blog/test-commander-phase-2-domain-agnostic-by-design.mdx) as your two canonical references.

**Length**: 3,500–4,500 words. Long enough to be substantive; short enough that a reader can finish in one sitting.

**Structure**:

1. **Opening (2–3 paragraphs)**. What happened. The single most surprising or instructive thing. Use the language of the actual session — quote the user's question that triggered the pivot, if there was one.
2. **The plan came first** (or equivalent). Restate the scaffold the work followed. Tables work well here.
3. **Per-step narrative**. One `##` section per major step or theme. Mix code blocks, bullet lists, and prose. Code blocks should be **real code from the repo**, not paraphrased.
4. **Bugs and lessons inline**. When you describe a bug, give the broken code, the diagnosis, and the fix in that order. Then state the lesson in one bold sentence the reader can grep for later.
5. **Patterns settled** (or "Patterns worth carrying forward"). 5–8 bold-led patterns that are now stable in the project. Each is one paragraph naming the pattern and what it produces.
6. **Closing**. What "done" means at the close of this phase / what ships materially / what comes next. End on the next anchor.

**Tone**:

- First person plural for design decisions ("we caught", "we shipped").
- First person singular for things the agent specifically did ("I authored the bookstore narrative").
- Honest about mistakes. The bookstore narrative was a mistake; the post says so directly. Avoid the corporate-blog tone where everything was always going perfectly. The discipline of the project is admitting when things went wrong.
- **No emojis anywhere** (AGENTS.md rule, applies to content too).
- Pull lessons from the project's `planning/plan.md` Lessons learned subsection where it exists. Do not duplicate the plan; quote sparingly and link out.

**Code blocks**:

- Use language hints: `python`, `yaml`, `markdown`, `ts`, `sh`.
- Show the broken version above the fix when documenting a bug — the diff is the lesson.
- Prefer real symbol names and paths from the repo over abstract examples.

## The hardcoded-count problem

This is the operational gotcha. `tests/e2e/blog.spec.ts` has multiple assertions that **must** be updated when a post lands. There are five places, and missing any one of them turns CI red.

When adding a post that becomes the new newest:

1. **Blog index post count and newest title**

   ```ts
   test("blog index lists all NN posts newest first", async ({ page }) => {
     // ...
     const titles = await page.getByRole("heading", { level: 2 }).allTextContents();
     expect(titles[0]).toMatch(/<new newest post title regex>/i);
     expect(titles).toHaveLength(NN);
   });
   ```

   Bump `NN`, update the test description ("eleven" → "twelve"), and update the title regex.

2. **Home Latest-writing newest title**

   ```ts
   const titles = await section.getByRole("heading", { level: 3 }).allTextContents();
   expect(titles).toHaveLength(3);
   expect(titles[0]).toMatch(/<new newest post title regex>/i);
   ```

   The home page renders the top 3 newest. Update only the title regex (length stays 3).

3. **RSS slug list** — add the new slug at the **top** of the array (newest first):

   ```ts
   for (const slug of [
     "<new-slug-here>",
     "<previous-newest-slug>",
     // ... existing slugs
   ]) {
     expect(body).toContain(`/blog/${slug}/`);
   }
   ```

4. **JSON Feed length and newest URL**

   ```ts
   expect(body.items).toHaveLength(NN);
   expect(body.items[0].url).toContain("/blog/<new-newest-slug>/");
   ```

   Bump `NN` and update the newest URL.

5. **Sitemap slug list** — same as RSS, add the new slug at the top:

   ```ts
   for (const slug of [
     "<new-slug-here>",
     "<previous-newest-slug>",
     // ...
   ]) {
     expect(body).toContain(`/blog/${slug}`);
   }
   ```

**Search shortcut.** Before authoring the post body, run this command to enumerate every place that hardcodes the previous newest post's slug or the previous post count:

```sh
grep -rnE 'building-test-commander-plan-driven-tdd-session|toHaveLength\([0-9]+\)' tests/ app/ lib/ 2>/dev/null
```

(Substitute the actual previous-newest slug for `building-test-commander-plan-driven-tdd-session`.) The grep should return only `tests/e2e/blog.spec.ts` hits. If it returns more, update those too.

## CHANGELOG entry

Open `CHANGELOG.md` and insert a new `Added` bullet at the top of the `Unreleased / Added` section. The convention is **one very long bullet per post** — a single paragraph that captures the post's coverage in enough detail that a future reader can decide whether to read the full post.

Pattern (extracted from existing entries):

```text
- New blog post: **<exact post title>** (`content/blog/<slug>.mdx`). <Long-form
  description of what the post covers: the major themes, the bugs and lessons
  catalogued, the patterns named, the closing thesis>. Categories: <comma-list>.
  Dated <YYYY-MM-DD> and sorts as the new newest post (above <previous-newest-slug>).
  Per-post OG image will regenerate on next `make build` via the `prebuild` script.
  Blog index, home Latest writing, RSS, JSON Feed, and sitemap iteration in
  `tests/e2e/blog.spec.ts` updated from <N> -> <N+1> posts and new newest title.
```

The entry can be 200–400 words. It is the post's index entry inside the project's own changelog; future maintenance work will rely on it.

## CI flow

The single command you must prove green before pushing is `make ci`. From the repo root:

```sh
cd ~/projects/testingwithagents
make ci
```

`make ci` runs (in order): `lint`, `typecheck`, `unit`, `build`, `e2e`, `a11y`, `linkcheck`. The order matters — the e2e and a11y suites run against the static `out/` directory produced by `build`.

Expected output for a clean run:

```text
144 passed (20.5s)               # e2e
54 passed (31.8s)                # a11y
547 Total, 520 OK, 0 Errors      # linkcheck (lychee — lychee prints its own emoji decorations)
Local CI complete.
```

Counts will shift slightly as the site grows. The signal you need is **zero errors** across all phases.

**Common failure modes:**

- **`toHaveLength(N)` mismatch** in `tests/e2e/blog.spec.ts` — the hardcoded count is one short of reality. Update it.
- **Newest title regex misses** — the home or blog-index assertion still matches the previous newest. Update the regex to the new title.
- **Schema rejection at build** — frontmatter has an unknown category or a missing required field. The build output names the file and the field; fix and re-run.
- **`prebuild` OG generation fails** — usually a missing slug in `lib/content/blog.ts` or a malformed frontmatter. Check the `scripts/generate-og-images.tsx` output near the top of `make build`.
- **Lychee link-check error** — you linked something that 404s, or you wrote a link with a relative path that broke. Fix the link or add it to the lychee exclude list (`lychee.toml`) only if it's a known false positive.

If `make ci` is red, fix and re-run until green. Do not push red.

## Commit and push

The repo's convention (visible in `git log`) is:

- Single descriptive commit per published post.
- Subject: `Publish "<exact post title>"`. The quotes go around the human-readable title.
- Body: multi-paragraph, describes the post's coverage, the test updates, and the local CI confirmation. The Phase 2 post's commit message is a good template — pull it up with `git log -1 --format=fuller c57775b` for reference.
- **No `Co-Authored-By: Claude` line**. The site's commits are attributed to the repo owner; the agent's role is documented in `AGENTS.md`, not per-commit.
- Use a HEREDOC for the commit message so multi-line formatting survives:

  ```sh
  git add CHANGELOG.md content/blog/<slug>.mdx public/og/blog/<slug>.png tests/e2e/blog.spec.ts
  git commit -m "$(cat <<'EOF'
  Publish "<exact post title>"

  <paragraph 1: what the post covers in 2-3 sentences>

  <paragraph 2: the bugs / lessons / patterns documented>

  <paragraph 3: housekeeping — categories, date, OG image, test count bump>

  Local CI green: 144 e2e + 54 a11y + 547-URL link check, zero errors.
  EOF
  )"
  git push origin main
  ```

The OG image lives at `public/og/blog/<slug>.png` after `make build`. Include it in the staged files even though it's auto-generated — it's a build artifact the post needs at runtime.

## Step-by-step checklist

Use this sequence end-to-end:

1. **Confirm the date.** Run `date -u +%Y-%m-%d`. That is `publishedAt`.
2. **Pick the slug.** Kebab-case. Match the eventual filename. Keep ≤ 60 chars if possible. Test that no existing post uses it: `ls content/blog/ | grep <slug>`.
3. **Identify the previous newest post**. `ls -t content/blog/ | head -3` then read each `publishedAt`. The newest is the one your new post will displace at the top.
4. **Inventory hardcoded references to the previous newest** — `grep -rn '<previous-newest-slug>' tests/ app/ lib/`. Expect hits in `tests/e2e/blog.spec.ts`. Note them.
5. **Author `content/blog/<slug>.mdx`** with the frontmatter and body per the conventions above. 3,500–4,500 words for project-log posts.
6. **Update `tests/e2e/blog.spec.ts`** (five places — see "The hardcoded-count problem" above):
   - Blog index test description + post count + newest title regex
   - Home Latest-writing newest title regex
   - RSS slug list (add new slug at top)
   - JSON Feed length + newest URL
   - Sitemap slug list (add new slug at top)
7. **Update `CHANGELOG.md`** — new `Added` bullet at top of `Unreleased`, per the pattern above.
8. **Run `make ci`** from the repo root. If red, fix and re-run. Do not proceed until green.
9. **Verify the OG image landed** — `ls public/og/blog/<slug>.png`. It should exist after `make build`. If missing, the prebuild script logged the failure earlier in the `make ci` output.
10. **Stage the right files**:

    ```sh
    git add CHANGELOG.md \
            content/blog/<slug>.mdx \
            public/og/blog/<slug>.png \
            tests/e2e/blog.spec.ts
    ```

    (Stage other files only if you genuinely modified them — never `git add -A`.)
11. **Commit with HEREDOC message** per the convention above. Subject `Publish "<exact post title>"`.
12. **Push** to `origin main`. The site rebuilds via Amplify automatically.

## What NOT to do

- Do not edit `app/blog/page.tsx`, `app/blog/[slug]/page.tsx`, `app/feed.json`, `app/rss.xml`, `app/sitemap.ts`, or `lib/content/blog.ts` unless you are explicitly changing the schema or the rendering. The defaults regenerate every post automatically.
- Do not hand-author OG images. The prebuild script handles them. If a post needs a special cover image, set `coverImage` in the frontmatter and place the asset under `public/blog/<slug>/`.
- Do not skip `make ci`. The hardcoded-count assertions are the project's safety net against the post-publishing footgun. Push only after green.
- Do not amend an existing post's `publishedAt` to reorder posts. Use `updatedAt` for revisions; `publishedAt` is the immutable original date.
- Do not write a "preview" or "stub" post. A post lands complete or not at all. Drafting belongs in a local file outside `content/blog/` until the body is ready.
- Do not invent emojis, decorative dividers, or non-standard frontmatter fields. The schema rejects them at build.

## When this runbook is wrong

If you follow this runbook and CI still goes red, **update the runbook in the same PR that fixes the underlying drift**. The runbook is preventative documentation; it is only useful while it matches reality. The discipline mirrors the Test Commander project's own "Sub-step lesson capture" convention — write the lesson down where the next operator will find it.
